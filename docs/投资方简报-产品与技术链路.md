# 家厘 — 产品与技术链路说明（面向投资方）

> 本文说明「用户如何在聊天中沉淀家庭记录与提醒」「系统如何保证时间与事实一致」「数据如何闭环到下一次对话」，便于理解产品机制与技术护城河，而非 API 细节。

---

## 1. 产品定位（一句话）

**以自然语言对话为入口的家庭事务助手**：把碎片信息结构化写入「家庭时间线 / 待办提醒」，并在后续对话中基于同一套数据库作答，避免「AI 说一套、App 里另一套」。

---

## 2. 用户能感知到的主路径

| 环节 | 用户侧体验 | 业务含义 |
|------|------------|----------|
| **聊天** | 像和朋友说话一样交代一件事（健康、缴费、证件、孩子日程等） | 低摩擦输入，无需填表 |
| **记录** | 在「记录」页看到家庭事件、提醒卡片 | 可核对、可编辑、可完成勾选 |
| **提醒** | 系统通知在约定时间触发 | 从「说过就算」到「到时真响铃」 |
| **再聊天** | 问「下周还有什么事」时，助手与列表一致 | 对话与数据同源，建立长期使用信任 |

---

## 3. 端到端链路（聊天 → 记录 → 提醒 → 回复）

### 3.1 整体流程（业务描述）

1. **用户发送一条消息**后，客户端将消息写入云端会话，并拉取家庭成员、近期家庭记录等上下文。
2. **意图判断**：系统先判断本轮是闲聊、查日程/历史，还是**需要写入记录或财务**，以及是否缺少关键信息（例如周期订阅但未确认日期）。缺信息时**不自动瞎写库**，而是让助手追问——降低「胡编日期」的产品与合规风险。
3. **结构化抽取**：在判定「值得落库」且槽位齐全时，专门调用一次大模型做**受约束的 JSON 抽取**（事件、提醒、可选的财务流水），与闲聊回复所用提示词分离，提高**字段级准确率**。
4. **写入策略（两阶段）**  
   - **先**：把与事实强相关的**家庭事件**等写入数据库（用户很快能在记录里看到脉络）。  
   - **后**：**提醒条目与财务流水**在生成**用户可见的助手回复**之后统一写入，并刷新本地通知调度。  
   这样缩短用户感知的「卡住时间」，同时让「助手嘴里说的」与「即将写入库里的」来自同一批计算结果。
5. **助手主回复**：主对话模型根据系统提示（含「本轮将写入数据库的事实」摘要）生成自然语言回复，**禁止与即将写入的数据矛盾**——用于向用户口述确认。
6. **记录页 / 提醒页**：均直接读写同一云端数据库（按家庭维度隔离），**无单独同步层**；用户在页面上修改时间会回写数据库，下一轮查询对话会读到最新值。

### 3.2 对投资叙事友好的要点

- **「对话即数据」**：不是纯聊天机器人，而是**对话驱动的家庭结构化数据沉淀**。
- **「同源事实」**：展示给用户的确认文案来自与写库相同的抽取与规则输出，降低幻觉导致的信任崩塌。
- **「克制写入」**：缺关键信息时不自动补全敏感日期，用追问代替臆造，有利于长期留存与口碑。

## 4. 关键技术设计（简述）

### 4.1 意图与抽取分离

- **意图分类**：快速判断场景（闲聊 / 查询 / 记录 / 取消提醒等），决定走不走抽取、是否允许写库。  
- **抽取模型**：专用 Prompt 与 JSON Schema，输出事件、提醒、流水等结构，再经**产品规则清洗**（例如禁止无依据的续费日、报销跟进模板等）。  

价值：**把「懂人话」与「写对表」拆成两步**，比单轮端到端生成 JSON 更可控。

### 4.2 时间与时区（家庭场景硬需求）

- 业务上统一按**北京时间（Asia/Shanghai）**理解「今天、明天、X 点半」等口语。  
- **一次性提醒**：区分**事项到点时刻**（`event_date`）与**通知响铃时刻**（`remind_at`），默认「提前若干分钟」等规则由代码固化，避免模型算错加减法。  
- **周期提醒（周/年等）**：入库保存周期规则与锚点时间；本地通知层按未来多期展开调度。  

价值：**时间管理是可审计的规则代码 + 数据库**，不仅依赖模型自由发挥。

### 4.3 家庭视角与表述规范

- 写入家庭的**标题/描述**会做「去第一人称、对齐成员姓名」等处理，保证**全家列表可读**、减少指代歧义。  
- 对话侧可注入「记录人 / 听者视角」等规则，减少家庭多角色使用时的误解。

### 4.4 通知与数据一致

- 未完成提醒会注册为系统**本地通知**；调度策略与库里的 `remind_at` 等字段对齐（一次性与周期策略有所区分）。  
- 用户完成或取消提醒后，数据状态变化可驱动**重新注册通知**，避免陈旧铃响。

---

## 5. 数据与安全（高层）

- **存储**：家庭、成员、会话、消息、家庭事件、提醒、财务流水等存放在 **Supabase（PostgreSQL）**，按 `family_id` 等边界做访问隔离；具体策略以后端 **RLS（行级安全）** 与业务规则为准。  
- **模型调用**：当前对话与抽取使用 **DeepSeek** API（密钥由客户端构建配置注入，生产环境建议迁移至服务端代理以降低泄露面——属后续工程硬化项）。  
- **隐私叙事**：家庭数据用于本家庭产品与提醒，不将聊天内容用于无关 third-party 广告profile（具体以正式隐私政策为准）。

---

## 6. 可扩展方向（ roadmap 叙述素材）

- **服务端编排**：意图、抽取、写库统一走自有机房 / 云函数，便于审计、AB、与更多模型路由。  
- **多模态**：语音已在工程上有对接能力；图片/单据可沿「抽取 → 结构化 → 同库」路径扩展。  
- **家庭图谱深化**：在现有成员与 kinship 规则上增强关系网络，支撑更复杂的「谁的事、谁买单、谁陪诊」等场景。  
- **开放能力**：日历导出、与家人共享视图的只读链接等，均建立在**已结构化的 `family_events` / `reminders`** 之上。
